== Overview

The application `lab-2-2-app` is stuck in `Pending` state. Unlike the previous lab, the PVC exists but it is also stuck in `Pending`. The PVC requests a StorageClass that does not exist on the cluster. Your mission is to fix the PVC so it can bind and the pod can start.

== Hints

[%collapsible]
.Hint 1 -- Where to look
====
The PVC references a StorageClass named `gold-storage` that does not exist on this cluster. You need to find out what StorageClasses are available and update the PVC accordingly.
====

[%collapsible]
.Hint 2 -- What to change
====
Run `oc get storageclass` to see the available StorageClasses. The default StorageClass (marked with `(default)`) is what you should use. You cannot edit a PVC's StorageClass directly -- you need to delete and recreate the PVC.
====

[%collapsible]
.Hint 3 -- Almost there
====
Delete the existing PVC, then create a new one with the same name but using the cluster's default StorageClass (or omit the `storageClassName` field entirely to use the default).
====

== Fix the Issue

Replace the PVC `lab-2-2-data` with one that uses an available StorageClass. You will need to delete the broken PVC and create a corrected one.

[%collapsible]
.Reveal Solution (Instructor Reference Only)
====
Delete the broken PVC:

[.console-input]
[source,bash,subs="attributes+"]
----
oc delete pvc lab-2-2-data -n {user}-lab
----

Create a corrected PVC without specifying a StorageClass (uses the default):

[.console-input]
[source,bash,subs="attributes+"]
----
cat <<'EOF' | oc apply -n {user}-lab -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: lab-2-2-data
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
EOF
----

Delete the pending pod to trigger rescheduling:

[.console-input]
[source,bash,subs="attributes+"]
----
oc delete pod -l app=lab-2-2-app -n {user}-lab
----
====
