Networking failures are among the hardest to diagnose because the pods involved are often in `Running` state. The problem is not that containers cannot start, but that traffic cannot reach them, or they cannot reach other services. In this module, you will work through four networking failure scenarios that cover the most common connectivity problems in OpenShift.

== Learning objectives

By the end of this module, you will be able to:

* Diagnose a service with no endpoints caused by a label selector mismatch
* Identify and fix a misconfigured Route that references the wrong service
* Resolve a DNS failure caused by using a short service name across namespaces
* Identify a NetworkPolicy that is blocking traffic and create a policy to allow it

== Background: OpenShift networking fundamentals

Traffic flows in OpenShift through a layered model:

* *Service*: A stable virtual IP that load-balances traffic to matching pods. The Service uses a `selector` to find pods by their labels. If no pods match the selector, the Service has no endpoints and traffic is dropped.
* *Route*: Exposes a Service outside the cluster via an externally accessible hostname. A Route references a specific Service and port. If the Service name or port is wrong, the Route returns `503 Service Unavailable`.
* *DNS*: OpenShift runs CoreDNS to resolve service names. Within the same namespace, a service is reachable by its name alone. Across namespaces, the fully qualified domain name (FQDN) `<service>.<namespace>.svc.cluster.local` is required.
* *NetworkPolicy*: A Kubernetes resource that controls which pods can communicate with each other. In namespaces where a NetworkPolicy exists, only explicitly permitted traffic is allowed.

== Exercise 1: Fix the selector mismatch

=== The scenario

Jordan reports that the `backend` service in `acme-networking` is unreachable even though the backend pods appear to be running. This is a classic selector mismatch.

=== Investigate

. Switch to the module namespace:
+
[source,bash]
----
oc project acme-networking
----

. Check pod status. Notice all pods show `Running`:
+
[source,bash]
----
oc get pods
----
+
Expected output:
+
----
NAME                         READY   STATUS    RESTARTS   AGE
backend-5c6d7e8f9-a1b2c      1/1     Running   0          15m
frontend-6d7e8f9a0-d3e4f     1/1     Running   0          15m
----
+
Pods are healthy. The problem must be in the network layer, not the application.

. Check the endpoints for the `backend` service:
+
[source,bash]
----
oc get endpoints
----
+
Expected output:
+
----
NAME       ENDPOINTS   AGE
backend    <none>      15m
frontend   10.128.2.45:8080   15m
----
+
The `backend` service has `<none>` endpoints. No pods are registered to receive traffic from this service, even though the backend pod is running.

. Describe the `backend` service to see its selector:
+
[source,bash]
----
oc describe service backend
----
+
Expected output (relevant section):
+
----
Selector:  app=backend-v2
----

. Check the labels on the actual backend pod:
+
[source,bash]
----
oc get pods --show-labels -l app=backend
----
+
Expected output:
+
----
NAME                     READY   STATUS    RESTARTS   AGE   LABELS
backend-5c6d7e8f9-a1b2c  1/1     Running   0          15m   app=backend,deployment=backend-5c6d7e8f9
----
+
The service selector is `app=backend-v2` but the pod has label `app=backend`. The selector does not match any pod, so the service has no endpoints.

// TODO: Add screenshot showing oc get endpoints with <none> and oc describe service showing the mismatched selector
image::module05-selector-mismatch.png[Terminal output showing oc get endpoints with none and oc describe service with wrong selector value,link=self,window=blank,width=700,title="Service Selector Mismatch - No Endpoints"]

=== Fix

Patch the service to use the correct selector:

[source,bash]
----
oc patch service backend -n acme-networking \
  -p '{"spec":{"selector":{"app":"backend"}}}'
----

=== Verify

. Confirm the service now has endpoints:
+
[source,bash]
----
oc get endpoints backend
----
+
Expected output:
+
----
NAME      ENDPOINTS           AGE
backend   10.128.2.46:8080    15m
----

. Test connectivity from the frontend pod to the backend service:
+
[source,bash]
----
oc exec -l app=frontend -- curl -s -o /dev/null -w "%{http_code}\n" http://backend:8080/health
----
+
Expected output:
+
----
200
----

* `oc get endpoints backend` shows a pod IP in the ENDPOINTS column
* `curl` from the frontend pod to `backend:8080` returns `200`

== Exercise 2: Fix the misconfigured Route

=== The scenario

The `storefront` application has running pods and a working service, but external users cannot reach it through the Route. The Route returns `503 Service Unavailable`.

=== Investigate

. Get the Route hostname:
+
[source,bash]
----
oc get route storefront-route
----
+
Expected output:
+
----
NAME               HOST/PORT                                                    PATH   SERVICES          PORT   TERMINATION   WILDCARD
storefront-route   storefront-route-acme-networking.{openshift_cluster_ingress_domain}          shop-svc   8080   edge          None
----

. Test the Route:
+
[source,bash]
----
curl -s -o /dev/null -w "%{http_code}\n" \
  https://storefront-route-acme-networking.{openshift_cluster_ingress_domain}
----
+
Expected output:
+
----
503
----

. Describe the Route to see which service it targets:
+
[source,bash]
----
oc describe route storefront-route
----
+
Expected output (relevant section):
+
----
Labels:     app=storefront
To:         Service with weight 100 (100%) -> shop-svc
            (svc not found)
----
+
The Route targets a service named `shop-svc`, but that service does not exist (marked as `svc not found`).

. Check what services actually exist in the namespace:
+
[source,bash]
----
oc get services
----
+
Expected output:
+
----
NAME         TYPE        CLUSTER-IP       EXTERNAL-IP   PORT(S)    AGE
backend      ClusterIP   172.30.45.67     <none>        8080/TCP   15m
frontend     ClusterIP   172.30.78.90     <none>        8080/TCP   15m
storefront   ClusterIP   172.30.12.34     <none>        8080/TCP   15m
----
+
The correct service name is `storefront`, not `shop-svc`. Alex used the wrong service name when creating the Route.

=== Fix

Patch the Route to reference the correct service:

[source,bash]
----
oc patch route storefront-route -n acme-networking \
  -p '{"spec":{"to":{"name":"storefront"}}}'
----

=== Verify

. Test the Route again:
+
[source,bash]
----
curl -s -o /dev/null -w "%{http_code}\n" \
  https://storefront-route-acme-networking.{openshift_cluster_ingress_domain}
----
+
Expected output:
+
----
200
----

. Confirm the Route now shows the service as found:
+
[source,bash]
----
oc describe route storefront-route | grep "To:"
----
+
Expected output:
+
----
To:         Service with weight 100 (100%) -> storefront
----

* Route description shows `storefront` service with no `(svc not found)` warning
* `curl` to the Route hostname returns `200`

== Exercise 3: Fix the DNS failure

=== The scenario

The `reporter` application collects metrics and sends them to a metrics collector service that runs in a different namespace (`acme-monitoring`). The pod is in `CrashLoopBackOff` because it cannot resolve the service hostname.

=== Investigate

. Check the `reporter` pod logs:
+
[source,bash]
----
oc logs -l app=reporter
----
+
Expected output:
+
----
[INFO]  Reporter starting...
[INFO]  Connecting to metrics collector at metrics-collector:9091
[ERROR] getaddrinfo ENOTFOUND metrics-collector
[ERROR] Failed to connect to metrics service. Exiting.
----
+
The application is trying to resolve `metrics-collector` as a short hostname. This only works for services in the same namespace.

. Test DNS resolution from inside the pod:
+
[source,bash]
----
oc exec -l app=reporter -- nslookup metrics-collector
----
+
Expected output:
+
----
Server:         172.30.0.10
Address:        172.30.0.10#53

** server can't find metrics-collector: NXDOMAIN
----
+
The short name `metrics-collector` does not resolve. The service is in the `acme-monitoring` namespace, not `acme-networking`.

. Confirm the service exists in the `acme-monitoring` namespace:
+
[source,bash]
----
oc get service metrics-collector -n acme-monitoring
----
+
Expected output:
+
----
NAME                TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
metrics-collector   ClusterIP   172.30.56.78    <none>        9091/TCP   30d
----

. Verify the FQDN resolves correctly:
+
[source,bash]
----
oc exec -l app=reporter -- nslookup metrics-collector.acme-monitoring.svc.cluster.local
----
+
Expected output:
+
----
Server:         172.30.0.10
Address:        172.30.0.10#53

Name:   metrics-collector.acme-monitoring.svc.cluster.local
Address: 172.30.56.78
----
+
The fully qualified domain name resolves. The fix is to update the application's `METRICS_HOST` environment variable to use the FQDN.

NOTE: OpenShift DNS resolution follows this priority: short name resolves within the same namespace, `<name>.<namespace>` does not work, and `<name>.<namespace>.svc.cluster.local` is the required FQDN for cross-namespace resolution.

=== Fix

Update the `reporter` deployment to use the FQDN for the metrics service:

[source,bash]
----
oc set env deployment/reporter \
  METRICS_HOST=metrics-collector.acme-monitoring.svc.cluster.local \
  -n acme-networking
----

=== Verify

. Watch the new pod start:
+
[source,bash]
----
oc get pods -l app=reporter -w
----
+
Expected output:
+
----
NAME                       READY   STATUS    RESTARTS   AGE
reporter-8a9b0c1d2-e3f4g   1/1     Running   0          15s
----
+
Press `Ctrl+C` to stop watching.

. Confirm the environment variable is correctly set:
+
[source,bash]
----
oc exec -l app=reporter -- env | grep METRICS_HOST
----
+
Expected output:
+
----
METRICS_HOST=metrics-collector.acme-monitoring.svc.cluster.local
----

* Pod `reporter` status is `Running`
* `METRICS_HOST` is set to the FQDN
* `nslookup metrics-collector.acme-monitoring.svc.cluster.local` resolves from inside the pod

== Exercise 4: Fix the NetworkPolicy

=== The scenario

The `api` application needs to connect to the `database` service, but the connection is timing out even though both pods are running and the service selector is correct. Jordan suspects a NetworkPolicy is blocking the traffic.

=== Investigate

. Confirm both pods are running and the service has endpoints:
+
[source,bash]
----
oc get pods
oc get endpoints database
----
+
Expected output:
+
----
NAME                         READY   STATUS    RESTARTS   AGE
api-3b4c5d6e7-f8g9h          1/1     Running   0          15m
database-9a0b1c2d3-i4j5k     1/1     Running   0          15m

NAME       ENDPOINTS            AGE
database   10.128.2.50:5432     15m
----
+
Pods are running and the service has endpoints. The problem is at the network layer.

. Test connectivity from the `api` pod to the `database` service:
+
[source,bash]
----
oc exec -l app=api -- curl -s --connect-timeout 5 http://database:5432
----
+
Expected output:
+
----
curl: (28) Connection timed out after 5000 milliseconds
----
+
The connection times out, which is the characteristic symptom of a NetworkPolicy drop. A connection refused error would come from a healthy but unexpected port; a timeout means the packet is silently dropped.

. Check for NetworkPolicies in the namespace:
+
[source,bash]
----
oc get networkpolicy -n acme-networking
----
+
Expected output:
+
----
NAME                  POD-SELECTOR    AGE
database-isolation    app=database    15m
----

. Describe the NetworkPolicy to see what traffic it permits:
+
[source,bash]
----
oc describe networkpolicy database-isolation
----
+
Expected output:
+
----
Name:         database-isolation
Namespace:    acme-networking
Spec:
  PodSelector: app=database
  Allowing ingress traffic:
    To Port: <any> (traffic allowed to all ports)
    From:
      PodSelector: role=monitoring
  Not affecting egress traffic
  Policy Types: Ingress
----
+
The NetworkPolicy allows ingress to `database` pods only from pods with label `role=monitoring`. Check what label the `api` pod has:

. Check the `api` pod labels:
+
[source,bash]
----
oc get pod -l app=api --show-labels
----
+
Expected output:
+
----
NAME                     READY   STATUS    RESTARTS   AGE   LABELS
api-3b4c5d6e7-f8g9h      1/1     Running   0          15m   app=api,role=api
----
+
The `api` pod has `role=api`. The NetworkPolicy only permits `role=monitoring`. Traffic from the `api` pod is blocked.

=== Fix

Create a new NetworkPolicy that permits ingress to the `database` pods from the `api` pods:

[source,bash]
----
cat <<EOF | oc apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-database
  namespace: acme-networking
spec:
  podSelector:
    matchLabels:
      app: database
  ingress:
  - from:
    - podSelector:
        matchLabels:
          role: api
  policyTypes:
  - Ingress
EOF
----

NOTE: The original `database-isolation` policy is left in place. Multiple NetworkPolicies targeting the same pods are combined with a logical OR: traffic is allowed if it satisfies any one of the matching policies.

=== Verify

. Test connectivity from the `api` pod to the `database` service again:
+
[source,bash]
----
oc exec -l app=api -- curl -s --connect-timeout 5 http://database:5432
----
+
The connection now succeeds (response will vary depending on the database type; what matters is that the connection is no longer timing out).

. Confirm both NetworkPolicies exist:
+
[source,bash]
----
oc get networkpolicy -n acme-networking
----
+
Expected output:
+
----
NAME                    POD-SELECTOR    AGE
allow-api-to-database   app=database    30s
database-isolation      app=database    15m
----

. Confirm all pods in the namespace are running:
+
[source,bash]
----
oc get pods -n acme-networking
----
+
Expected output:
+
----
NAME                         READY   STATUS    RESTARTS   AGE
api-3b4c5d6e7-f8g9h          1/1     Running   0          18m
backend-5c6d7e8f9-a1b2c      1/1     Running   0          18m
database-9a0b1c2d3-i4j5k     1/1     Running   0          18m
frontend-6d7e8f9a0-d3e4f     1/1     Running   0          18m
reporter-8a9b0c1d2-e3f4g     1/1     Running   0          5m
storefront-2e3f4a5b6-g7h8i   1/1     Running   0          18m
----

* `oc exec -l app=api -- curl http://database:5432` no longer times out
* Two NetworkPolicies exist for the `database` pod selector
* All six pods in `acme-networking` show `Running`

== Troubleshooting

**Issue**: `oc get endpoints <service>` still shows `<none>` after patching the selector

**Solution**:

. Verify the patch was applied: `oc describe service backend | grep Selector`
. Confirm pod labels match exactly (case-sensitive): `oc get pods --show-labels`
. Check if any pods have the label but are in a non-Running state: `oc get pods -l app=backend`
. If pods are running and labels match, wait a few seconds. Endpoint updates can take 5-10 seconds to propagate.

---

**Issue**: Route returns `503` after patching to the correct service name

**Solution**:

. Confirm the service exists and has a matching port: `oc describe service storefront`
. Verify the Route's `targetPort` matches a port defined in the service spec
. Check for router pod errors: `oc get pods -n openshift-ingress`
. Wait 15-30 seconds after patching; the router needs to reload its configuration.

---

**Issue**: `nslookup <fqdn>` fails inside the pod

**Solution**:

. First confirm CoreDNS is running: `oc get pods -n openshift-dns`
. Test basic DNS resolution: `oc exec -l app=reporter -- nslookup kubernetes.default.svc.cluster.local`
. If basic resolution fails, restart the CoreDNS pods: `oc rollout restart deployment/dns-default -n openshift-dns`
. Verify the target service exists in the remote namespace: `oc get svc -n acme-monitoring`

---

**Issue**: Connection still times out after applying the new NetworkPolicy

**Solution**:

. Verify the new NetworkPolicy was created with the correct pod selector: `oc describe networkpolicy allow-api-to-database`
. Confirm the `api` pod label matches the NetworkPolicy's `from.podSelector`: `oc get pod -l app=api --show-labels`
. Wait 10-15 seconds. NetworkPolicy changes are propagated to all nodes by the network plugin and may take a moment.
. If the timeout persists, check for an egress NetworkPolicy on the `api` namespace that may be blocking outbound traffic.

---

**Issue**: Unsure if a timeout is caused by a NetworkPolicy or something else

**Solution**:

* A NetworkPolicy drop always produces a connection *timeout* (the packet is silently discarded). A *connection refused* error means the packet reached the destination but was rejected at the application or OS level. A timeout pointing to a ClusterIP service almost always indicates a NetworkPolicy.
* Verify with: `oc get networkpolicy -A | grep <namespace>` to check for policies across all namespaces.

== Learning outcomes

By completing this module, you now understand:

* How to use `oc get endpoints` to determine whether a Service is selecting any pods, and how to trace a selector mismatch to its root cause
* How a Route's `spec.to.name` must exactly match an existing Service name, and how `(svc not found)` in Route describe output identifies the problem
* That OpenShift DNS short names only resolve within the same namespace, and cross-namespace services require the FQDN `<service>.<namespace>.svc.cluster.local`
* How to read a NetworkPolicy to understand which traffic it permits, and how to create a complementary policy to allow additional traffic
* The difference between a connection timeout (NetworkPolicy drop) and a connection refused (application rejection)

== Module summary

You have resolved four networking failures in ACME Corp's `acme-networking` namespace.

**What you accomplished:**

* Fixed a `backend` Service with a selector mismatch that produced no endpoints
* Fixed a `storefront-route` Route pointing to a non-existent service name
* Fixed a `reporter` pod in `CrashLoopBackOff` by updating its `METRICS_HOST` env var to use the cross-namespace FQDN
* Fixed a `database` connection timeout by creating a NetworkPolicy to allow ingress from `api` pods

**Key takeaways:**

* Always check `oc get endpoints` when an application cannot reach a service. No endpoints means no pods match the selector. Use `oc get pods --show-labels` to compare.
* A Route returning `503` with `(svc not found)` means the Route's target service name does not match any service in the namespace.
* DNS short names work within a namespace. For cross-namespace service discovery, always use the FQDN: `<service>.<namespace>.svc.cluster.local`.
* Connection timeouts (not refused) to a ClusterIP are almost always caused by a NetworkPolicy. Check `oc get networkpolicy -n <namespace>` first.
