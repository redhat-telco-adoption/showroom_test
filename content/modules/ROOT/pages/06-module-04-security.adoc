Security failures in OpenShift manifest in three distinct ways: application code that cannot access the Kubernetes API, pods blocked from starting by Security Context Constraint violations, and containers stuck in `ImagePullBackOff` because registry credentials are missing. In this module, you will diagnose and resolve all three.

== Learning objectives

By the end of this module, you will be able to:

* Diagnose and fix RBAC errors that prevent a ServiceAccount from accessing Kubernetes API resources
* Identify a Security Context Constraint violation that prevents pods from being created
* Diagnose `ImagePullBackOff` errors caused by missing image pull credentials
* Use `oc auth can-i`, `oc get scc`, and `oc describe replicaset` as part of security troubleshooting

== Background: OpenShift security model

OpenShift enforces security at multiple layers:

* *RBAC (Role-Based Access Control)*: Controls which users and ServiceAccounts can perform actions on API resources. Roles define permissions; RoleBindings assign roles to subjects.
* *Security Context Constraints (SCCs)*: Control what a pod is allowed to do at the OS level, including which user IDs it can run as, whether it can use privileged capabilities, and what volume types it can mount. SCCs are enforced at pod admission time.
* *Image pull credentials*: Required to pull container images from private registries. Stored as Secrets of type `kubernetes.io/dockerconfigjson` and linked to ServiceAccounts.

== Exercise 1: Fix the RBAC error

=== The scenario

The `config-manager` service is a background process that reads ConfigMaps from the namespace to dynamically update application configuration. Jordan confirms it has been crashing since Alex changed its ServiceAccount. The pod starts but crashes immediately.

=== Investigate

. Switch to the module namespace:
+
[source,bash]
----
oc project acme-security
----

. Check pod status:
+
[source,bash]
----
oc get pods
----
+
Expected output:
+
----
NAME                              READY   STATUS             RESTARTS   AGE
config-manager-6b7c8d9e0-f3g4h    0/1     CrashLoopBackOff   5          10m
image-server-4c5d6e7f8-w2x1y      0/1     ImagePullBackOff   0          10m
webapp-7d8e9f0a1-b2c3d             0/1     Pending            0          10m
----

. Check the logs for `config-manager`:
+
[source,bash]
----
oc logs -l app=config-manager
----
+
Expected output:
+
----
[INFO]  Config Manager starting...
[INFO]  Connecting to Kubernetes API...
[ERROR] Failed to list ConfigMaps: configmaps is forbidden: User
        "system:serviceaccount:acme-security:config-sa" cannot list
        resource "configmaps" in API group "" in the namespace "acme-security"
[ERROR] Startup failed. Exiting with code 1.
----
+
The application uses its ServiceAccount (`config-sa`) to call the Kubernetes API. That ServiceAccount does not have permission to list ConfigMaps.

. Confirm the ServiceAccount exists but has no relevant RoleBinding:
+
[source,bash]
----
oc get serviceaccount config-sa -n acme-security
----
+
[source,bash]
----
oc get rolebinding -n acme-security
----
+
Expected output for rolebinding:
+
----
NAME                    ROLE                   AGE
system:deployers        Role/system:deployer   30d
system:image-builders   Role/system:image-builder  30d
----
+
No RoleBinding assigns any role to `config-sa`.

. Use `oc auth can-i` to confirm the ServiceAccount cannot list ConfigMaps:
+
[source,bash]
----
oc auth can-i list configmaps \
  --as=system:serviceaccount:acme-security:config-sa \
  -n acme-security
----
+
Expected output:
+
----
no
----

=== Fix

Grant the `config-sa` ServiceAccount read access to resources in the namespace using the built-in `view` ClusterRole:

[source,bash]
----
oc create rolebinding config-sa-view \
  --clusterrole=view \
  --serviceaccount=acme-security:config-sa \
  -n acme-security
----

Restart the deployment to create a new pod with the updated permissions:

[source,bash]
----
oc rollout restart deployment/config-manager -n acme-security
----

=== Verify

. Confirm the permission is now granted:
+
[source,bash]
----
oc auth can-i list configmaps \
  --as=system:serviceaccount:acme-security:config-sa \
  -n acme-security
----
+
Expected output:
+
----
yes
----

. Watch the new pod start:
+
[source,bash]
----
oc get pods -l app=config-manager -w
----
+
Expected output:
+
----
NAME                             READY   STATUS    RESTARTS   AGE
config-manager-8f9a0b1c2-h5i6j   1/1     Running   0          15s
----
+
Press `Ctrl+C` to stop watching.

* `oc auth can-i list configmaps --as=system:serviceaccount:acme-security:config-sa` returns `yes`
* Pod `config-manager` status is `Running`

== Exercise 2: Fix the Security Context Constraint violation

=== The scenario

The `webapp` deployment shows no running pods at all, even though the Deployment resource exists. Jordan notes that Alex copied the deployment manifest from an older Docker Compose setup. OpenShift's Security Context Constraints prevent pods from running as root by default.

=== Investigate

. Check that the deployment exists but has no pods:
+
[source,bash]
----
oc get deployment webapp
----
+
Expected output:
+
----
NAME     READY   UP-TO-DATE   AVAILABLE   AGE
webapp   0/1     0            0           10m
----
+
The deployment has 0 pods ready and 0 up-to-date. A pod should exist but does not. Check the ReplicaSet for more detail.

. Describe the ReplicaSet owned by the webapp Deployment:
+
[source,bash]
----
oc describe replicaset -l app=webapp
----
+
Look at the *Conditions* section:
+
----
Conditions:
  Type             Status  Reason
  ----             ------  ------
  ReplicaFailure   True    FailedCreate
----
+
And in the Events section:
+
----
Events:
  Type     Reason        Age                  From                   Message
  ----     ------        ----                 ----                   -------
  Warning  FailedCreate  10m (x10 over 10m)   replicaset-controller  Error creating: pods "webapp-7d8e9f0a1-" is forbidden: unable to validate against any security context constraint: [spec.containers[0].securityContext.runAsUser: Invalid value: 0: must be in the ranges: [1000660000, 1000669999]]
----
+
Pod creation is rejected at admission because the deployment requests `runAsUser: 0` (root), which the `restricted` SCC does not permit.

// TODO: Add screenshot of oc describe replicaset output showing the FailedCreate event with SCC violation
image::module04-scc-violation-event.png[Terminal output showing oc describe replicaset with FailedCreate event and SCC violation message,link=self,window=blank,width=700,title="SCC Violation - Pod Creation Rejected"]

. Confirm the problematic setting in the deployment:
+
[source,bash]
----
oc get deployment webapp -o yaml | grep -A 5 securityContext
----
+
Expected output:
+
----
      securityContext:
        runAsUser: 0
----
+
The deployment explicitly requests root user (UID 0). OpenShift's `restricted` SCC only allows UIDs in the namespace's assigned range.

NOTE: OpenShift automatically assigns each namespace a unique UID range from the `openshift.io/sa.scc.uid-range` annotation. Applications should not specify a `runAsUser` and instead let OpenShift assign a UID from this range. This is the correct approach for most applications.

. Check your namespace's assigned UID range:
+
[source,bash]
----
oc describe namespace acme-security | grep uid-range
----
+
Expected output:
+
----
Annotations: openshift.io/sa.scc.uid-range=1000660000/10000
----

=== Fix

Remove the `runAsUser: 0` setting from the deployment. OpenShift will assign a UID from the namespace's allocated range:

[source,bash]
----
oc patch deployment webapp -n acme-security --type=json \
  -p='[{"op":"remove","path":"/spec/template/spec/securityContext/runAsUser"}]'
----

=== Verify

. Watch a pod get created and start:
+
[source,bash]
----
oc get pods -l app=webapp -w
----
+
Expected output:
+
----
NAME                      READY   STATUS              RESTARTS   AGE
webapp-2e3f4a5b6-c7d8e    0/1     ContainerCreating   0          5s
webapp-2e3f4a5b6-c7d8e    1/1     Running             0          10s
----
+
Press `Ctrl+C` to stop watching.

. Confirm the pod is running as a non-root UID assigned by OpenShift:
+
[source,bash]
----
oc exec -l app=webapp -- id
----
+
Expected output (UID will be from your namespace range):
+
----
uid=1000660000(1000660000) gid=0(root) groups=0(root),1000660000
----

* Pod `webapp` status is `Running`
* Pod UID is from the namespace range, not `0`

== Exercise 3: Fix the image pull secret

=== The scenario

The `image-server` pod is stuck in `ImagePullBackOff`. The deployment references an image in a private container registry that requires authentication credentials. No pull secret has been configured for this namespace.

=== Investigate

. Describe the `image-server` pod:
+
[source,bash]
----
oc describe pod -l app=image-server
----
+
Look at the Events section:
+
----
Events:
  Type     Reason     Age                  From               Message
  ----     ------     ----                 ----               -------
  Normal   Scheduled  10m                  default-scheduler  Successfully assigned acme-security/image-server-4c5d6e7f8-w2x1y to worker-2
  Normal   Pulling    10m (x4 over 10m)    kubelet            Pulling image "registry.example.com/acme/image-server:v2.1"
  Warning  Failed     10m (x4 over 10m)    kubelet            Failed to pull image "registry.example.com/acme/image-server:v2.1": rpc error: code = Unknown desc = failed to resolve reference "registry.example.com/acme/image-server:v2.1": unexpected status code 401 Unauthorized
  Warning  BackOff    9m (x6 over 10m)     kubelet            Back-off pulling image "registry.example.com/acme/image-server:v2.1"
----
+
The registry returns `401 Unauthorized`, which means the pull is failing because no credentials are provided.

. Check whether any pull secrets are currently linked to the `default` ServiceAccount in this namespace:
+
[source,bash]
----
oc describe serviceaccount default -n acme-security
----
+
Expected output:
+
----
Name:                default
Namespace:           acme-security
Image pull secrets:  default-dockercfg-xxxxx
Mountable secrets:   default-dockercfg-xxxxx
                     default-token-xxxxx
----
+
Only the internal cluster registry pull secret (`default-dockercfg-xxxxx`) is linked. There are no credentials for `registry.example.com`.

=== Fix

. Create a Secret with the registry credentials:
+
[source,bash]
----
oc create secret docker-registry registry-creds \
  --docker-server=registry.example.com \
  --docker-username=acme-ci \
  --docker-password=r3g1stryP@ss \
  -n acme-security
----

. Link the secret to the `default` ServiceAccount so all pods in the namespace can use it to pull images:
+
[source,bash]
----
oc secrets link default registry-creds --for=pull -n acme-security
----

. Trigger a new pod rollout so the new pod picks up the pull secret:
+
[source,bash]
----
oc rollout restart deployment/image-server -n acme-security
----

=== Verify

. Watch the new pod pull the image and start:
+
[source,bash]
----
oc get pods -l app=image-server -w
----
+
Expected output:
+
----
NAME                           READY   STATUS              RESTARTS   AGE
image-server-4c5d6e7f8-w2x1y   0/1     ImagePullBackOff    0          12m
image-server-6f7a8b9c0-x3y4z   0/1     ContainerCreating   0          8s
image-server-6f7a8b9c0-x3y4z   1/1     Running             0          20s
image-server-4c5d6e7f8-w2x1y   0/1     Terminating         0          12m
----
+
Press `Ctrl+C` to stop watching.

. Confirm the pull secret is now linked to the `default` ServiceAccount:
+
[source,bash]
----
oc describe serviceaccount default -n acme-security | grep registry-creds
----
+
Expected output:
+
----
Image pull secrets:  default-dockercfg-xxxxx
                     registry-creds
----

. Confirm all three security module applications are running:
+
[source,bash]
----
oc get pods -n acme-security
----
+
Expected output:
+
----
NAME                              READY   STATUS    RESTARTS   AGE
config-manager-8f9a0b1c2-h5i6j    1/1     Running   0          5m
image-server-6f7a8b9c0-x3y4z      1/1     Running   0          1m
webapp-2e3f4a5b6-c7d8e             1/1     Running   0          3m
----

* All three pods show `Running` status with `1/1` READY
* `oc auth can-i list configmaps --as=system:serviceaccount:acme-security:config-sa` returns `yes`
* `oc describe serviceaccount default` shows `registry-creds` in the pull secrets list

== Troubleshooting

**Issue**: Pod still crashes after creating the RoleBinding

**Solution**:

. Verify the RoleBinding references the correct namespace and ServiceAccount:
+
[source,bash]
----
oc describe rolebinding config-sa-view -n acme-security
----
. Confirm the RoleBinding subject matches the ServiceAccount name exactly (case-sensitive)
. Restart the deployment to force a new pod: `oc rollout restart deployment/config-manager -n acme-security`
. Check updated logs: `oc logs -l app=config-manager`

---

**Issue**: `oc patch` to remove `runAsUser` fails with "remove operation does not apply"

**Solution**:

* The JSON Patch `remove` operation requires the path to exist. If `securityContext.runAsUser` is nested differently, use `oc edit` instead.
* Use `oc edit deployment webapp -n acme-security` and manually delete the `runAsUser: 0` line.
* After saving, the ReplicaSet controller will retry pod creation automatically.

---

**Issue**: Webapp pod starts but the application logs show "permission denied" for certain operations

**Solution**:

* The application may have been written assuming root privileges and uses operations that require them.
* Check the logs: `oc logs -l app=webapp`
* Identify which operations fail and whether they can be rewritten to work as a non-root user.
* If the application genuinely requires a specific UID (but not root), use `runAsUser` with a value in the namespace's allowed UID range. Find the range: `oc describe namespace acme-security | grep uid-range`

---

**Issue**: Pod still shows `ImagePullBackOff` after linking the pull secret

**Solution**:

. The existing pod does not automatically retry with new credentials. Trigger a fresh rollout:
+
[source,bash]
----
oc rollout restart deployment/image-server -n acme-security
----
. Verify the new pod uses the updated ServiceAccount: `oc describe pod -l app=image-server | grep pull`
. If the error continues, double-check the credentials: `oc create secret docker-registry test-creds --docker-server=registry.example.com --docker-username=acme-ci --docker-password=r3g1stryP@ss --dry-run=client -o yaml`

---

**Issue**: `oc secrets link` succeeds but the new pod still cannot pull the image

**Solution**:

* The pod may not be using the `default` ServiceAccount. Check which ServiceAccount the deployment uses:
+
[source,bash]
----
oc get deployment image-server -o jsonpath='{.spec.template.spec.serviceAccountName}'
----
* If it uses a custom ServiceAccount, link the secret to that ServiceAccount instead:
+
[source,bash]
----
oc secrets link <service-account-name> registry-creds --for=pull -n acme-security
----

== Learning outcomes

By completing this module, you now understand:

* How to identify a RBAC error from application logs (`is forbidden: User "system:serviceaccount:..."`) and verify permissions with `oc auth can-i`
* How to grant a ServiceAccount namespace-scoped API permissions using `oc create rolebinding` with built-in ClusterRoles like `view`
* Why OpenShift rejects pods with `runAsUser: 0` and how the `restricted` SCC protects the cluster by enforcing non-root execution
* How to diagnose a missing pull secret from `oc describe pod` events and fix it using `oc create secret docker-registry` and `oc secrets link`
* The difference between a pod that crashes after starting (RBAC), a pod that is never created (SCC), and a pod stuck pulling an image (pull secret)

== Module summary

You have resolved three security failures in ACME Corp's `acme-security` namespace.

**What you accomplished:**

* Diagnosed and fixed a `config-manager` pod in `CrashLoopBackOff` caused by a missing RBAC RoleBinding for its ServiceAccount
* Diagnosed and fixed a `webapp` deployment with no pods due to an SCC violation (`runAsUser: 0` blocked by the `restricted` SCC)
* Diagnosed and fixed an `image-server` pod in `ImagePullBackOff` by creating a registry pull secret and linking it to the default ServiceAccount

**Key takeaways:**

* RBAC errors appear in application logs as `is forbidden: User "system:serviceaccount:..."`. Use `oc auth can-i --as=system:serviceaccount:<ns>:<sa>` to verify and confirm the fix.
* An SCC violation prevents pods from being created entirely. Check `oc describe replicaset` when `oc get pods` shows no pods for a deployment that should have them.
* OpenShift enforces non-root execution by default. Remove explicit `runAsUser: 0` from deployments and let OpenShift assign a UID from the namespace range.
* `ImagePullBackOff` with a `401 Unauthorized` error always means missing or incorrect registry credentials. After linking the pull secret, trigger a rollout restart to create a pod that uses the new credentials.
