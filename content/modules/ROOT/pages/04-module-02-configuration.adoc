Jordan's follow-up arrives in your inbox fifteen minutes after standup:

____
"I've identified the first batch of broken apps. They're all in the `acme-config` namespace. The `frontend`, `backend`, and `api` services are all down. Alex told me it's probably 'a config thing'. Can you investigate and get them running?"

Configuration errors are among the most common reasons applications fail on OpenShift. In this module, you will investigate three broken deployments that cannot start because of missing or misconfigured configuration resources.

== Learning objectives

By the end of this module, you will be able to:

* Diagnose pod failures caused by missing ConfigMaps using pod events and describe output
* Diagnose pod failures caused by missing Secrets
* Identify and fix invalid environment variable key references in a deployment
* Create ConfigMaps and Secrets using `oc create` to restore failing applications

== Background: OpenShift configuration resources

OpenShift provides two resource types for injecting configuration into pods:

* *ConfigMap*: Stores non-sensitive configuration data as key-value pairs or file content. Use for application settings, feature flags, and connection strings.
* *Secret*: Stores sensitive data such as passwords, tokens, and certificates. Values are base64-encoded and subject to RBAC access controls.

Both resources can be consumed by pods in three ways:

* *envFrom*: All keys from a ConfigMap or Secret are injected as environment variables
* *valueFrom*: A specific key from a ConfigMap or Secret is injected as a named environment variable
* *Volume mount*: Content is mounted as files in the container filesystem

If a pod references a ConfigMap or Secret that does not exist, or references a key that does not exist within the resource, the pod will not start. The events section of `oc describe` reveals the exact error.

== Exercise 1: Fix the missing ConfigMap

=== The scenario

Jordan's note identifies the `frontend` service as the first broken app. You switch to the `acme-config` namespace and start your investigation.

=== Investigate

. Switch to the module namespace:
+
[source,bash]
----
oc project acme-config
----

. Check pod status across the namespace:
+
[source,bash]
----
oc get pods
----
+
Expected output:
+
----
NAME                         READY   STATUS                       RESTARTS   AGE
api-5b6f7d8c9-r7t2v          0/1     CreateContainerConfigError   0          8m
backend-7f9dc8b4c-p3k9n      0/1     Pending                      0          8m
frontend-6d8b9c5f4-xq2m7     0/1     Pending                      0          8m
----
+
Three broken pods, three different scenarios. Start with `frontend`.

. Describe the `frontend` pod to find the root cause:
+
[source,bash]
----
oc describe pod -l app=frontend
----
+
Scroll to the *Events* section at the bottom of the output:
+
----
Events:
  Type     Reason    Age                  From               Message
  ----     ------    ----                 ----               -------
  Normal   Scheduled 8m                   default-scheduler  Successfully assigned acme-config/frontend-6d8b9c5f4-xq2m7 to worker-1
  Warning  Failed    8m (x5 over 8m)      kubelet            Error: configmap "app-config" not found
----
+
The pod has been scheduled to a node, but the kubelet cannot start the container because the required ConfigMap does not exist.

// TODO: Add screenshot of oc describe output showing the Events section with configmap not found error
image::module02-configmap-not-found-event.png[Terminal output showing oc describe pod with the Events section highlighting the configmap not found error,link=self,window=blank,width=700,title="oc describe - ConfigMap Not Found Event"]

. Examine the deployment to understand what the `frontend` app expects:
+
[source,bash]
----
oc get deployment frontend -o yaml | grep -A 5 envFrom
----
+
Expected output:
+
----
        envFrom:
        - configMapRef:
            name: app-config
----
+
The deployment uses `envFrom.configMapRef` to inject all keys from a ConfigMap named `app-config` as environment variables. That ConfigMap does not exist yet.

=== Fix

. Create the missing ConfigMap with the application's required configuration:
+
[source,bash]
----
oc create configmap app-config \
  --from-literal=APP_ENV=production \
  --from-literal=APP_PORT=8080 \
  --from-literal=LOG_LEVEL=info \
  -n acme-config
----

. Confirm the ConfigMap was created with the correct keys:
+
[source,bash]
----
oc get configmap app-config -o yaml
----
+
Expected output:
+
----
apiVersion: v1
data:
  APP_ENV: production
  APP_PORT: "8080"
  LOG_LEVEL: info
kind: ConfigMap
metadata:
  name: app-config
  namespace: acme-config
----

=== Verify

. Watch the pod status update:
+
[source,bash]
----
oc get pods -l app=frontend -w
----
+
Within 30 seconds, the pod should transition to `Running`:
+
----
NAME                         READY   STATUS              RESTARTS   AGE
frontend-6d8b9c5f4-xq2m7     0/1     ContainerCreating   0          9m
frontend-6d8b9c5f4-xq2m7     1/1     Running             0          9m
----
+
Press `Ctrl+C` to stop watching.

* Pod `frontend` status is `Running`
* Pod `READY` shows `1/1`

== Exercise 2: Fix the missing Secret

=== The scenario

The `backend` service is also stuck in `Pending`. Jordan's note says it needs database credentials. The error pattern is the same as the ConfigMap case, but this time the missing resource is a Secret.

=== Investigate

. Describe the `backend` pod:
+
[source,bash]
----
oc describe pod -l app=backend
----
+
Events section:
+
----
Events:
  Type     Reason    Age                From     Message
  ----     ------    ----               ----     -------
  Normal   Scheduled 9m                 default-scheduler  Successfully assigned acme-config/backend-7f9dc8b4c-p3k9n to worker-2
  Warning  Failed    9m (x5 over 9m)    kubelet  Error: secret "db-credentials" not found
----
+
Same pattern as the ConfigMap error: the resource is referenced in the deployment but has not been created.

. Confirm what the deployment expects:
+
[source,bash]
----
oc get deployment backend -o yaml | grep -A 5 secretRef
----
+
Expected output:
+
----
        envFrom:
        - secretRef:
            name: db-credentials
----

NOTE: Secrets are referenced in pod specs the same way as ConfigMaps, using `secretRef` instead of `configMapRef`. The `envFrom.secretRef` pattern injects all keys from the Secret as environment variables.

=== Fix

. Create the missing Secret with the database credentials:
+
[source,bash]
----
oc create secret generic db-credentials \
  --from-literal=DB_USERNAME=acmeuser \
  --from-literal=DB_PASSWORD=Sup3rS3cr3t! \
  --from-literal=DB_HOST=postgres.acme-config.svc.cluster.local \
  -n acme-config
----

. Verify the Secret was created (values are stored base64-encoded):
+
[source,bash]
----
oc get secret db-credentials -o yaml
----
+
Expected output (values are encoded):
+
----
apiVersion: v1
data:
  DB_HOST: cG9zdGdyZXMuYWNtZS1jb25maWcuc3ZjLmNsdXN0ZXIubG9jYWw=
  DB_PASSWORD: U3VwM3JTM2NyM3Qh
  DB_USERNAME: YWNtZXVzZXI=
kind: Secret
metadata:
  name: db-credentials
  namespace: acme-config
----

IMPORTANT: Never store actual production credentials in workshop exercises or commit them to version control. In production environments, use a dedicated secrets management solution such as Red Hat OpenShift Secrets Store CSI Driver or an external vault.

=== Verify

. Watch the pod come back up:
+
[source,bash]
----
oc get pods -l app=backend -w
----
+
Expected output:
+
----
NAME                        READY   STATUS    RESTARTS   AGE
backend-7f9dc8b4c-p3k9n     1/1     Running   0          10m
----
+
Press `Ctrl+C` to stop watching.

* Pod `backend` status is `Running`
* Pod `READY` shows `1/1`

== Exercise 3: Fix the invalid environment variable reference

=== The scenario

The `api` service shows `CreateContainerConfigError` rather than `Pending`. This is a meaningful difference. Jordan confirms: "Alex says he created the ConfigMap for the api app this time. But it's still broken."

`CreateContainerConfigError` indicates that the ConfigMap or Secret *exists*, but there is a problem with how it is referenced. A specific key is missing or misspelled.

=== Investigate

. Describe the `api` pod:
+
[source,bash]
----
oc describe pod -l app=api
----
+
Events section:
+
----
Events:
  Type     Reason    Age                 From      Message
  ----     ------    ----                ----      -------
  Normal   Scheduled 10m                 default-scheduler  Successfully assigned acme-config/api-5b6f7d8c9-r7t2v to worker-1
  Warning  Failed    10m (x5 over 10m)   kubelet   Error: couldn't find key DB_HOST in ConfigMap acme-config/api-config
----
+
The ConfigMap `api-config` exists, but the pod references a key called `DB_HOST` which is not present in the ConfigMap.

. Check the ConfigMap to see what keys are actually defined:
+
[source,bash]
----
oc get configmap api-config -o yaml
----
+
Expected output:
+
----
apiVersion: v1
data:
  DATABASE_HOST: postgres.acme-config.svc.cluster.local
  DATABASE_NAME: acmedb
  DATABASE_PORT: "5432"
kind: ConfigMap
metadata:
  name: api-config
  namespace: acme-config
----
+
The ConfigMap uses `DATABASE_HOST` but the deployment references `DB_HOST`. This is a naming mismatch.

. Confirm the mismatch by checking the deployment configuration:
+
[source,bash]
----
oc get deployment api -o yaml | grep -A 8 valueFrom
----
+
Expected output:
+
----
        env:
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              key: DB_HOST
              name: api-config
----
+
The deployment uses `valueFrom.configMapKeyRef` to inject a single key as the `DB_HOST` environment variable, but specifies `key: DB_HOST` which does not exist in `api-config`.

=== Fix

You have two options to resolve this mismatch:

* *Option A*: Edit the deployment to reference the correct key name (recommended when the ConfigMap naming is correct)
* *Option B*: Edit the ConfigMap to add the expected key (use when the deployment key name is the established convention)

In this case, the ConfigMap uses a descriptive naming convention (`DATABASE_HOST`). Update the deployment to use the correct key.

. Edit the deployment to correct the key reference:
+
[source,bash]
----
oc edit deployment api -n acme-config
----
+
Find the `configMapKeyRef` section and change `key: DB_HOST` to `key: DATABASE_HOST`:
+
----
        env:
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              key: DATABASE_HOST    # changed from DB_HOST
              name: api-config
----
+
Save and exit the editor (`Escape`, then `:wq`, then `Enter` for vi). OpenShift automatically rolls out a new pod.
+
Alternatively, use `oc patch` for a non-interactive fix:
+
[source,bash]
----
oc patch deployment api -n acme-config --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/0/env/0/valueFrom/configMapKeyRef/key","value":"DATABASE_HOST"}]'
----

=== Verify

. Watch the new pod roll out:
+
[source,bash]
----
oc get pods -l app=api -w
----
+
Expected output:
+
----
NAME                     READY   STATUS                       RESTARTS   AGE
api-5b6f7d8c9-r7t2v      0/1     CreateContainerConfigError   0          12m
api-7c8d9e0f1-n4w6x      0/1     ContainerCreating            0          5s
api-7c8d9e0f1-n4w6x      1/1     Running                      0          8s
api-5b6f7d8c9-r7t2v      0/1     Terminating                  0          12m
----
+
Press `Ctrl+C` to stop watching.

. Confirm the environment variable is correctly injected in the running container:
+
[source,bash]
----
oc exec deployment/api -- env | grep DB_HOST
----
+
Expected output:
+
----
DB_HOST=postgres.acme-config.svc.cluster.local
----

. Confirm all three applications are now running:
+
[source,bash]
----
oc get pods -n acme-config
----
+
Expected output:
+
----
NAME                         READY   STATUS    RESTARTS   AGE
api-7c8d9e0f1-n4w6x          1/1     Running   0          1m
backend-7f9dc8b4c-p3k9n      1/1     Running   0          3m
frontend-6d8b9c5f4-xq2m7     1/1     Running   0          5m
----

* All three pods show `Running` status with `1/1` READY
* `oc exec deployment/api -- env | grep DB_HOST` returns the correct hostname value

== Troubleshooting

**Issue**: Pod remains `Pending` after creating the ConfigMap or Secret

**Solution**:

. Verify the resource was created in the correct namespace:
+
[source,bash]
----
oc get configmap -n acme-config
oc get secret -n acme-config
----
. Confirm the resource name exactly matches what the deployment references. Names are case-sensitive.
. If the names match but the pod is still pending, trigger a new rollout:
+
[source,bash]
----
oc rollout restart deployment/<name> -n acme-config
----

---

**Issue**: `oc describe` shows "configmap not found" but `oc get configmap` shows it exists

**Solution**:

* You are likely in the wrong namespace. ConfigMaps are namespace-scoped resources.
* Verify your current namespace: `oc project`
* Check the namespace in the error message from `oc describe` and compare it to where you created the ConfigMap.

---

**Issue**: Pod transitions to `Running` briefly, then enters `CrashLoopBackOff`

**Solution**:

* The configuration error is resolved, but the application has a separate problem.
* Check current logs: `oc logs <pod-name> -n acme-config`
* Check previous logs for the crash reason: `oc logs <pod-name> --previous -n acme-config`
* The application may be failing because a configuration value is incorrect (for example, a database hostname that does not resolve).

---

**Issue**: `oc edit deployment` opens but navigating the YAML is difficult

**Solution**:

* In `vi`, type `/DB_HOST` and press `Enter` to search for the key
* Press `n` to jump to the next match
* Press `i` to enter insert mode and make your edit
* Press `Escape`, then type `:wq` and press `Enter` to save and exit
* Use the `oc patch` command shown in Exercise 3 as a non-interactive alternative

---

**Issue**: `oc patch` command fails with an array index error

**Solution**:

* The JSON path `/spec/template/spec/containers/0/env/0/...` assumes the first container and first env var. If there are multiple containers or env vars, the index may differ.
* Inspect the full deployment JSON to find the correct index:
+
[source,bash]
----
oc get deployment api -o json | python3 -m json.tool | grep -n "DB_HOST"
----
* Use `oc edit` for complex changes involving multiple containers or env vars.

== Learning outcomes

By completing this module, you now understand:

* How ConfigMaps and Secrets are referenced in OpenShift deployments using `envFrom.configMapRef`, `envFrom.secretRef`, and `valueFrom.configMapKeyRef`
* How to identify a missing ConfigMap or Secret error from pod events (`Error: configmap/secret not found`)
* The difference between a missing resource (`Pending`) and an invalid key reference (`CreateContainerConfigError`)
* How to create ConfigMaps and Secrets using `oc create` with `--from-literal` flags
* How to correct a key name mismatch using `oc edit` and `oc patch`
* How to verify configuration injection using `oc exec deployment/<name> -- env`

You have resolved three configuration failures in ACME Corp's `acme-config` namespace.

**What you accomplished:**

* Diagnosed and fixed a `frontend` pod blocked by a missing ConfigMap
* Diagnosed and fixed a `backend` pod blocked by a missing Secret
* Diagnosed and fixed an `api` pod in `CreateContainerConfigError` caused by an invalid ConfigMap key reference
* Used `oc describe`, `oc get -o yaml`, `oc exec`, `oc edit`, and `oc patch` to investigate and resolve each issue

**Key takeaways:**

* `Pending` pods with a missing ConfigMap or Secret always show `Error: configmap/secret not found` in events. The fix is to create the missing resource.
* `CreateContainerConfigError` means the resource exists but a key is missing or misspelled. Always check both the ConfigMap/Secret keys and the deployment's key references.
* Names and key names are case-sensitive. A mismatch like `DB_HOST` vs `DATABASE_HOST` will prevent the container from starting.
* `oc exec deployment/<name> -- env | grep <VAR>` is a fast way to confirm configuration was injected correctly after a fix.
