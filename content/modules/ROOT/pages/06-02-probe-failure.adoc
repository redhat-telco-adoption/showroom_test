= Lab 6.2: Probe Failure

:module_namespace: {ns_lifecycle}

[.label.label-yellow]#Intermediate# | *Persona:* Developer | *Category:* Application Lifecycle

== Overview

The application `lab-6-2-app` starts but never becomes Ready (shows `0/1`). The container is running but Kubernetes keeps restarting it because the readiness and liveness probes are misconfigured -- they check the wrong path and wrong port. Your mission is to fix the probe configuration.

== Learning objectives

* Identify readiness and liveness probe failures causing `0/1 Running` state using `oc describe pod` events
* Understand how misconfigured probe paths and ports prevent pods from becoming Ready
* Apply corrected probe configuration with the right path, port, and timing to restore the pod to `Ready` state

=== Diagnose

[tabs]
====
Web Console::
+
--
. Navigate to *Workloads -> Pods* in namespace `{module_namespace}`
. Click the `lab-6-2-app` pod
. Review the *Events* tab -- look for probe failure messages:
  * `Readiness probe failed: HTTP probe failed with statuscode: 404`
  * `Liveness probe failed: HTTP probe failed with statuscode: 404`
. Navigate to *Workloads -> Deployments*, click `lab-6-2-app`, and review the YAML
. Check the `readinessProbe` and `livenessProbe` configuration
--

Web Terminal (`oc`)::
+
--
[.console-input]
[source,bash,subs="attributes+"]
----
oc describe pod -l app=lab-6-2-app -n {module_namespace}
----

Look for probe failures in Events:

[.console-output]
[source,text]
----
Warning  Unhealthy  ...  Readiness probe failed: HTTP probe failed with statuscode: 404
Warning  Unhealthy  ...  Liveness probe failed: Get "http://10.128.x.x:9090/healthz": dial tcp 10.128.x.x:9090: connect: connection refused
----

Check the probe configuration:

[.console-input]
[source,bash,subs="attributes+"]
----
oc get deployment lab-6-2-app -n {module_namespace} -o yaml | grep -A 8 'Probe'
----

[.console-output]
[source,yaml]
----
readinessProbe:
  httpGet:
    path: /healthz
    port: 9090
  initialDelaySeconds: 1
livenessProbe:
  httpGet:
    path: /healthz
    port: 9090
  initialDelaySeconds: 1
----

The probes target path `/healthz` on port `9090`, but nginx serves on port `8080` and the path `/healthz` returns 404.
--
====

=== Hints

[%collapsible]
.Hint 1 -- Where to look
====
The probes check `/healthz` on port `9090`. The nginx container listens on port `8080` and does not have a `/healthz` endpoint. Both the path and port are wrong.
====

[%collapsible]
.Hint 2 -- What to change
====
The nginx container serves its default page at path `/` on port `8080`. Update both the readiness and liveness probes to use `path: /` and `port: 8080`. Also increase `initialDelaySeconds` to give the container time to start.
====

[%collapsible]
.Hint 3 -- Almost there
====
Edit the Deployment and change both probes: set `path` to `/`, `port` to `8080`, and `initialDelaySeconds` to `5`. Use `oc edit deployment` to make all changes at once.
====

== Exercise 2: Resolve the issue

=== Fix the issue

Update the readiness and liveness probes to use the correct path, port, and timing values.

[%collapsible]
.Reveal Solution (Instructor Reference Only)
====
[.console-input]
[source,bash,subs="attributes+"]
----
oc edit deployment lab-6-2-app -n {module_namespace}
----

Update both probes:

[source,yaml]
----
readinessProbe:
  httpGet:
    path: /
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
livenessProbe:
  httpGet:
    path: /
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 20
----

Or use patch:

[.console-input]
[source,bash,subs="attributes+"]
----
oc patch deployment lab-6-2-app -n {module_namespace} --type=json -p='[
  {"op": "replace", "path": "/spec/template/spec/containers/0/readinessProbe", "value": {"httpGet": {"path": "/", "port": 8080}, "initialDelaySeconds": 5, "periodSeconds": 10}},
  {"op": "replace", "path": "/spec/template/spec/containers/0/livenessProbe", "value": {"httpGet": {"path": "/", "port": 8080}, "initialDelaySeconds": 15, "periodSeconds": 20}}
]'
----
====
