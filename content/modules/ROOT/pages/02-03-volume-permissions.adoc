= Lab 2.3: Volume Permissions
include::_attributes.adoc[]

[.label.label-yellow]#Intermediate# | *Persona:* Platform Engineer | *Category:* Storage

== Overview

The application `lab-2-3-app` starts but fails to operate correctly. The container logs reveal `Permission denied` errors when trying to write to the mounted volume at `/data`. Your mission is to diagnose the volume ownership issue and fix the Deployment so the container process can write to the volume.

== Hints

[%collapsible]
.Hint 1 -- Where to look
====
The issue is about filesystem permissions on the mounted volume. When a PVC is first mounted, the filesystem is owned by root. If the container process runs as a non-root user, it cannot write to the volume.
====

[%collapsible]
.Hint 2 -- What needs to change
====
Kubernetes provides a `securityContext` field at the pod level that can set the group ownership of mounted volumes. Look into the `fsGroup` setting.
====

[%collapsible]
.Hint 3 -- Almost there
====
Add `securityContext.fsGroup: 1000` to the pod spec (not the container spec). This tells Kubernetes to change the group ownership of all mounted volumes to GID 1000, which matches the container user.
====

== Fix the Issue

Modify the Deployment's pod security context to ensure the mounted volume is accessible by the container process. You need to set the `fsGroup` field.

[%collapsible]
.Reveal Solution (Instructor Reference Only)
====
[.console-input]
[source,bash,subs="attributes+"]
----
oc patch deployment lab-2-3-app -n {user}-lab --type=json \
  -p='[{"op": "add", "path": "/spec/template/spec/securityContext", "value": {"fsGroup": 1000}}]'
----

Alternatively, edit the Deployment YAML directly:

[.console-input]
[source,bash,subs="attributes+"]
----
oc edit deployment lab-2-3-app -n {user}-lab
----

Add under `spec.template.spec`:

[source,yaml]
----
securityContext:
  fsGroup: 1000
----

This triggers a new rollout with the corrected volume permissions.
====
