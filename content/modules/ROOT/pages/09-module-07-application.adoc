Jordan's final message arrives:

____
"Last one! The `acme-application` namespace has three more broken apps. The `scheduler` keeps crashing, the `api-gateway` is unstable with high restarts, and `webapp-v2` won't start at all. Fix these and Alex's mess will be completely cleaned up before he gets back!"
____

The failures in this module are application-level: pods that crash because a required startup condition is not met, pods that are repeatedly killed because their health probes are misconfigured, and pods stuck because they reference a container image tag that does not exist. Each requires a slightly different diagnostic approach.

== Learning objectives

By the end of this module, you will be able to:

* Use `oc logs` and `oc logs --previous` to find the root cause of a `CrashLoopBackOff` and fix it
* Read liveness probe failure events in `oc describe pod` output and correct a misconfigured probe path
* Distinguish between an image pull authentication failure (Module 4) and an image tag not found error, and fix the latter with `oc set image`

== Background: Application-level failure patterns

Not all pod failures are caused by missing resources or misconfigurations in cluster infrastructure. Some failures are purely at the application layer:

* *CrashLoopBackOff*: The container starts, runs briefly, then exits with a non-zero code. Kubernetes retries with exponential backoff. The cause is always in the application logs.
* *Probe failures*: Kubernetes uses liveness and readiness probes to monitor container health. A liveness probe failure kills and restarts the container. A readiness probe failure removes the pod from service endpoints without killing it. Both are configured in the deployment spec and must match the application's actual endpoints and ports.
* *ImagePullBackOff (tag not found)*: The image name is correct and credentials are in place, but the specific tag referenced in the deployment does not exist in the registry. The fix is to update the image reference to a valid tag.

== Exercise 1: Fix the CrashLoopBackOff

=== The scenario

The `scheduler` service starts and then immediately exits. Jordan notes: "It's been crashing since Alex deployed it. Looks like he forgot to finish the configuration."

=== Investigate

. Switch to the module namespace:
+
[source,bash]
----
oc project acme-application
----

. Check pod status across the namespace:
+
[source,bash]
----
oc get pods
----
+
Expected output:
+
----
NAME                              READY   STATUS             RESTARTS      AGE
api-gateway-4b5c6d7e8-f9g0h       1/1     Running            11            15m
scheduler-6c7d8e9f0-i1j2k         0/1     CrashLoopBackOff   9             15m
webapp-v2-8e9f0a1b2-k3l4m         0/1     ImagePullBackOff   0             15m
----
+
Note that `api-gateway` is `Running` but with 11 restarts. That will be Exercise 2. Start with `scheduler`.

. Fetch the current logs. Because the container crashes quickly, use `--previous` to see the last crash:
+
[source,bash]
----
oc logs -l app=scheduler --previous
----
+
Expected output:
+
----
[INFO]  Scheduler service starting...
[INFO]  Reading configuration...
[ERROR] Required environment variable APP_MODE is not set.
[ERROR] Valid values: 'production', 'development'
[ERROR] Cannot start without APP_MODE. Exiting.
----
+
The application requires an `APP_MODE` environment variable to be set before it will start. Without it, the process exits with a non-zero code immediately.

. Confirm `APP_MODE` is not set in the deployment:
+
[source,bash]
----
oc set env deployment/scheduler --list -n acme-application
----
+
Expected output:
+
----
# scheduler, container scheduler
DB_HOST=postgres.acme-application.svc.cluster.local
DB_PORT=5432
----
+
`DB_HOST` and `DB_PORT` are present. `APP_MODE` is not.

=== Fix

Set the missing environment variable directly on the deployment:

[source,bash]
----
oc set env deployment/scheduler APP_MODE=production -n acme-application
----

=== Verify

. Watch the new pod start and remain stable:
+
[source,bash]
----
oc get pods -l app=scheduler -w
----
+
Expected output:
+
----
NAME                         READY   STATUS    RESTARTS   AGE
scheduler-2a3b4c5d6-e7f8g    1/1     Running   0          15s
----
+
Wait at least 60 seconds to confirm the restart count stays at 0. Press `Ctrl+C` to stop watching.

. Confirm the env var is set and the app started cleanly:
+
[source,bash]
----
oc logs -l app=scheduler
----
+
Expected output:
+
----
[INFO]  Scheduler service starting...
[INFO]  Reading configuration...
[INFO]  APP_MODE: production
[INFO]  Connecting to database at postgres.acme-application.svc.cluster.local:5432
[INFO]  Scheduler ready. Listening for jobs.
----

* Pod `scheduler` status is `Running` with `RESTARTS` at `0`
* Application logs show a clean startup sequence

== Exercise 2: Fix the probe failure

=== The scenario

The `api-gateway` pod shows `Running` but has accumulated 11 restarts in 15 minutes. This pattern indicates a liveness probe that is repeatedly failing and triggering container restarts. The pod is technically "up" between kills, which is why it shows `Running` rather than `CrashLoopBackOff`.

=== Investigate

. Describe the `api-gateway` pod and focus on the Events section:
+
[source,bash]
----
oc describe pod -l app=api-gateway
----
+
Events section:
+
----
Events:
  Type     Reason     Age                  From     Message
  ----     ------     ----                 ----     -------
  Warning  Unhealthy  40s (x22 over 14m)   kubelet  Liveness probe failed: HTTP probe failed with statuscode: 404
  Warning  Killing    35s (x11 over 14m)   kubelet  Container api-gateway failed liveness probe, will be restarted
----
+
The liveness probe is returning HTTP `404`. This means the probe is hitting a path that the application does not serve.

. Check the liveness probe configuration on the deployment:
+
[source,bash]
----
oc get deployment api-gateway -o yaml | grep -A 10 livenessProbe
----
+
Expected output:
+
----
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 15
            failureThreshold: 3
----
+
The probe checks `/healthz` on port 8080. Verify what the application actually exposes:

. Exec into the running container to check available health endpoints:
+
[source,bash]
----
oc exec -l app=api-gateway -- curl -s http://localhost:8080/healthz
----
+
Expected output:
+
----
404 page not found
----
+
[source,bash]
----
oc exec -l app=api-gateway -- curl -s http://localhost:8080/health
----
+
Expected output:
+
----
{"status":"ok","uptime":47}
----
+
The application exposes `/health`, not `/healthz`. The probe path has a typo.

NOTE: Liveness and readiness probes have different consequences. A *liveness* probe failure kills and restarts the container. A *readiness* probe failure only removes the pod from the service endpoints without restarting it. Both must match the application's actual health endpoint path and port.

=== Fix

Patch the deployment to correct the liveness probe path from `/healthz` to `/health`:

[source,bash]
----
oc patch deployment api-gateway -n acme-application --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/0/livenessProbe/httpGet/path","value":"/health"}]'
----

=== Verify

. Watch the rolling update complete:
+
[source,bash]
----
oc get pods -l app=api-gateway -w
----
+
Expected output:
+
----
NAME                           READY   STATUS              RESTARTS   AGE
api-gateway-4b5c6d7e8-f9g0h    1/1     Running             11         16m
api-gateway-9c0d1e2f3-n5o6p    0/1     ContainerCreating   0          5s
api-gateway-9c0d1e2f3-n5o6p    1/1     Running             0          12s
api-gateway-4b5c6d7e8-f9g0h    0/1     Terminating         11         16m
----
+
Press `Ctrl+C` to stop watching.

. Wait 60 seconds and confirm the new pod is not accumulating restarts:
+
[source,bash]
----
oc get pods -l app=api-gateway
----
+
Expected output:
+
----
NAME                          READY   STATUS    RESTARTS   AGE
api-gateway-9c0d1e2f3-n5o6p   1/1     Running   0          90s
----

. Confirm the probe now succeeds:
+
[source,bash]
----
oc describe pod -l app=api-gateway | grep -A 5 Liveness
----
+
Expected output (no Unhealthy events):
+
----
    Liveness:   http-get http://:8080/health delay=10s timeout=1s period=15s #success=1 #failure=3
----

* Pod `api-gateway` status is `Running` with `RESTARTS` remaining at `0` over 60 seconds
* No `Unhealthy` or `Killing` events appear in `oc describe pod` output

== Exercise 3: Fix the wrong image tag

=== The scenario

The `webapp-v2` pod is stuck in `ImagePullBackOff`. Alex was testing a pre-release version of the image and accidentally left that tag in the deployment manifest. The release tag that should be used is `v2.0`.

=== Investigate

. Describe the `webapp-v2` pod:
+
[source,bash]
----
oc describe pod -l app=webapp-v2
----
+
Events section:
+
----
Events:
  Type     Reason     Age                  From               Message
  ----     ------     ----                 ----               -------
  Normal   Scheduled  15m                  default-scheduler  Successfully assigned acme-application/webapp-v2-8e9f0a1b2-k3l4m to worker-1
  Normal   Pulling    14m (x4 over 15m)    kubelet            Pulling image "registry.example.com/acme/webapp:v2.1-beta"
  Warning  Failed     14m (x4 over 15m)    kubelet            Failed to pull image "registry.example.com/acme/webapp:v2.1-beta": rpc error: code = NotFound desc = failed to pull and unpack image "registry.example.com/acme/webapp:v2.1-beta": unexpected status code 404: manifest unknown
  Warning  BackOff    2m (x8 over 13m)     kubelet            Back-off pulling image "registry.example.com/acme/webapp:v2.1-beta"
----
+
The error is `manifest unknown` with HTTP `404`. This is different from the `401 Unauthorized` error in Module 4. The registry is reachable and credentials are valid, but the tag `v2.1-beta` does not exist.

. Confirm the image reference in the deployment:
+
[source,bash]
----
oc get deployment webapp-v2 -o jsonpath='{.spec.template.spec.containers[0].image}'
----
+
Expected output:
+
----
registry.example.com/acme/webapp:v2.1-beta
----

. Compare with the `webapp` deployment (the v1 version) to see how the image should be referenced:
+
[source,bash]
----
oc get deployment webapp -o jsonpath='{.spec.template.spec.containers[0].image}'
----
+
Expected output:
+
----
registry.example.com/acme/webapp:v2.0
----
+
The stable production image uses tag `v2.0`. Alex had updated the `webapp-v2` deployment to reference `v2.1-beta`, which has not been published to the registry yet.

TIP: When diagnosing `ImagePullBackOff`, look for the exact error in the pod events. `401 Unauthorized` means missing or wrong registry credentials (see Module 4). `404` with `manifest unknown` means the image tag does not exist in the registry.

=== Fix

Update the deployment to use the correct image tag:

[source,bash]
----
oc set image deployment/webapp-v2 \
  webapp-v2=registry.example.com/acme/webapp:v2.0 \
  -n acme-application
----

=== Verify

. Watch the new pod pull the corrected image and start:
+
[source,bash]
----
oc get pods -l app=webapp-v2 -w
----
+
Expected output:
+
----
NAME                           READY   STATUS              RESTARTS   AGE
webapp-v2-8e9f0a1b2-k3l4m      0/1     ImagePullBackOff    0          16m
webapp-v2-0b1c2d3e4-p7q8r      0/1     ContainerCreating   0          8s
webapp-v2-0b1c2d3e4-p7q8r      1/1     Running             0          18s
webapp-v2-8e9f0a1b2-k3l4m      0/1     Terminating         0          16m
----
+
Press `Ctrl+C` to stop watching.

. Confirm the correct image tag is now in use:
+
[source,bash]
----
oc get pod -l app=webapp-v2 -o jsonpath='{.items[0].spec.containers[0].image}'
----
+
Expected output:
+
----
registry.example.com/acme/webapp:v2.0
----

. Confirm all three application module pods are now running:
+
[source,bash]
----
oc get pods -n acme-application
----
+
Expected output:
+
----
NAME                          READY   STATUS    RESTARTS   AGE
api-gateway-9c0d1e2f3-n5o6p   1/1     Running   0          5m
scheduler-2a3b4c5d6-e7f8g     1/1     Running   0          8m
webapp-v2-0b1c2d3e4-p7q8r     1/1     Running   0          1m
----

* All three pods show `Running` status with `1/1` READY
* `webapp-v2` image references `v2.0`, not `v2.1-beta`
* `api-gateway` restart count remains at `0`

== Troubleshooting

**Issue**: `oc logs --previous` shows no output or "previous terminated container not found"

**Solution**:

* The pod may not have had a previous container yet (first crash hasn't completed). Wait for the pod to cycle once and try again.
* If the pod is in `Pending` state, there are no logs yet. Describe the pod instead: `oc describe pod -l app=scheduler`
* If the container exits too fast for logs to be written, the application may be crashing before initializing its logger. Check the exit code in `oc describe pod`: exit code `1` is a general error, exit code `2` is a misuse of shell built-ins, exit code `126/127` means the command or binary was not found.

---

**Issue**: Pod restarts stop but the restart count shown is high

**Solution**:

* A high `RESTARTS` count is historical. After the probe or crash fix, what matters is whether the count stays constant.
* Wait 2-3 minutes and check again: `oc get pods -l app=api-gateway`
* If the count is still climbing, the probe fix may not have been applied correctly. Verify with: `oc describe pod -l app=api-gateway | grep -A 8 Liveness`

---

**Issue**: `oc exec -- curl http://localhost:8080/health` fails with "connection refused"

**Solution**:

* The application may not be listening on port 8080. Check which port it actually uses:
+
[source,bash]
----
oc exec -l app=api-gateway -- ss -tlnp
----
* Look for a LISTEN entry and note the port number
* Verify the livenessProbe port in the deployment matches: `oc get deployment api-gateway -o yaml | grep -A 5 livenessProbe`

---

**Issue**: `oc set image` runs without error but the pod still uses the old image

**Solution**:

* The `oc set image` command updates the deployment spec. A rolling update is triggered automatically.
* Check that the rollout progressed: `oc rollout status deployment/webapp-v2 -n acme-application`
* If the rollout is stuck, describe the new pod to see why it is not starting: `oc describe pod -l app=webapp-v2`

---

**Issue**: `manifest unknown` error but you are sure the tag exists

**Solution**:

* Verify there is no typo in the image name or tag: `oc get deployment webapp-v2 -o jsonpath='{.spec.template.spec.containers[0].image}'`
* Check whether the registry URL is correct. Some registries are case-sensitive.
* Confirm the image is in the correct registry project or organization.
* If the tag was recently pushed, the registry cache may need a moment. Wait 30 seconds and trigger a rollout restart: `oc rollout restart deployment/webapp-v2 -n acme-application`

== Learning outcomes

By completing this module, you now understand:

* How to use `oc logs --previous` to retrieve the log output from a crashed container and identify why a `CrashLoopBackOff` occurs
* How liveness probe failures appear in `oc describe pod` events (`Unhealthy` and `Killing`), and how to confirm the correct probe path by using `oc exec` to test endpoints directly from inside the running container
* The distinction between `401 Unauthorized` (missing credentials) and `404 manifest unknown` (tag does not exist) in image pull errors, and how to use `oc set image` to correct an image reference
* How `oc set env --list` quickly shows what environment variables a deployment currently has configured
* Why a pod in `Running` state with a high and climbing restart count should be treated as broken, not healthy

== Module summary

You have resolved the final three application failures in ACME Corp's `acme-application` namespace, completing all of Alex's broken deployments.

**What you accomplished:**

* Fixed the `scheduler` pod in `CrashLoopBackOff` by setting the missing `APP_MODE=production` environment variable
* Fixed the `api-gateway` pod with repeated probe-triggered restarts by correcting the liveness probe path from `/healthz` to `/health`
* Fixed the `webapp-v2` pod in `ImagePullBackOff` by updating the image tag from the non-existent `v2.1-beta` to the stable `v2.0`

**Key takeaways:**

* `CrashLoopBackOff` always has a reason in the logs. `oc logs --previous` is the first command to run. The exit code in `oc describe pod` narrows the category: `1` is an application error, `127` means binary not found, `137` means OOMKilled (see Module 6).
* A pod in `Running` state with a climbing restart count has a liveness probe issue. The probe is killing a container that would otherwise stay up. Check `oc describe pod` events for `Unhealthy` and `Killing` entries.
* `manifest unknown` (404) means the image tag does not exist in the registry. Always confirm the exact tag with `oc get deployment -o jsonpath='{.spec.template.spec.containers[0].image}'` before attempting a fix.
