=== Switch to the Developer perspective

The OpenShift web console has two main perspectives:

* *Administrator* perspective: Full cluster access, ideal for platform engineers and cluster admins. Shows all namespaces, cluster operators, and infrastructure resources.
* *Developer* perspective: Application-focused view for developers. Topology view makes it easy to see application components and their connections.

. To switch to the *Developer* perspective, click the *Administrator* dropdown in the top-left corner and select *Developer*.

. In the *Developer* perspective, select the `{namespace}` project from the project selector at the top of the page.

. Click *Topology* in the left navigation menu.
+
You should see a topology map showing the deployed applications in the namespace. Several will show error indicators, which you will investigate in later modules.

. Click on one of the application nodes in the topology view to open its side panel.

== Exercise 3: Core diagnostic commands

The OpenShift CLI gives you the power to quickly inspect any resource in the cluster. These four commands form the foundation of all OpenShift troubleshooting. You will use them in every module of this workshop.

=== oc get pods

`oc get pods` is the first command you run when investigating a problem. It shows the status of all pods in the current namespace.

. List all pods in the workshop namespace:
+
[source,bash]
----
oc get pods -n {namespace}
----
+
Review the `STATUS` and `RESTARTS` columns. Pods with high restart counts or non-Running status need attention.

. For additional detail including the node each pod is running on:
+
[source,bash]
----
oc get pods -n {namespace} -o wide
----

Common pod status values and what they indicate:

[cols="1,2",options="header"]
|===
|Status |Meaning

|`Running`
|Pod is healthy and all containers are running

|`Pending`
|Pod is waiting to be scheduled or for a required resource to become available

|`CrashLoopBackOff`
|Container is repeatedly crashing after starting; check logs for the crash reason

|`ImagePullBackOff`
|Kubernetes cannot pull the container image; check image name and pull credentials

|`Error`
|Container exited with a non-zero exit code; check logs for details

|`OOMKilled`
|Container was terminated because it exceeded its memory limit

|`ContainerCreating`
|Pod is in the process of starting up (usually temporary)

|`Terminating`
|Pod is being deleted
|===

=== oc describe

`oc describe` is your most powerful diagnostic command. It shows detailed information about any resource, including configuration, recent events, and status conditions.

. Identify the name of a failing pod from the previous step, then describe it:
+
[source,bash]
----
oc describe pod <pod-name> -n {namespace}
----

. Pay particular attention to these sections in the describe output:
+
* *Events* (at the bottom of the output): Shows what OpenShift tried to do and any errors it encountered. This is often the fastest path to the root cause.
* *State*: The current and previous container state with exit codes and reasons
* *Conditions*: Pod-level readiness and scheduling conditions

TIP: The Events section in `oc describe` output is often the fastest way to identify why a pod is failing. Always scroll to the bottom and start there.

=== oc logs

`oc logs` retrieves the stdout output from a container, which is where most applications write their error messages.

include::partial$check-logs.adoc[]

. Try fetching logs from one of the failing pods:
+
[source,bash]
----
oc logs <pod-name> -n {namespace}
----

TIP: If a pod is in `CrashLoopBackOff`, the container restarts quickly. Add `--previous` to see the logs from before the last crash: `oc logs <pod-name> --previous -n {namespace}`

=== oc get events

Events provide a timeline of what happened in a namespace. They capture scheduler decisions, image pulls, volume mounts, and failures.

include::partial$check-events.adoc[]

. Check current events in the workshop namespace:
+
[source,bash]
----
oc get events -n {namespace} --sort-by='.lastTimestamp'
----

=== The diagnostic workflow

When you encounter a broken application, use this four-step sequence:

. Check pod status to identify failing pods:
+
[source,bash]
----
oc get pods -n {namespace}
----

. Describe the failing pod to see its events and conditions:
+
[source,bash]
----
oc describe pod <pod-name> -n {namespace}
----

. Check the pod logs for application-level errors:
+
[source,bash]
----
oc logs <pod-name> -n {namespace}
----

. Review namespace events for a full timeline:
+
[source,bash]
----
oc get events -n {namespace} --sort-by='.lastTimestamp'
----

This four-step sequence resolves the majority of OpenShift application failures. You will use it throughout this workshop.

== Troubleshooting

**Issue**: `oc login` fails with "Login failed (401 Unauthorized)"

**Solution**:

* Verify your username and password from the lab credentials panel on the right
* Confirm you are using the correct API URL: `{openshift_api_url}`
* Try the login command again after 30 seconds (the cluster API may be briefly unavailable)

---

**Issue**: `oc: command not found` in the web terminal

**Solution**:

. Refresh the browser and reopen the web terminal by clicking the `>_` icon
. The `oc` CLI is pre-installed in the OpenShift web terminal and should be available by default
. If the issue persists, open the terminal from the OCP Console tab and navigate to link:{openshift_console_url}[{openshift_console_url}^]

---

**Issue**: "Error from server (Forbidden)" when running `oc` commands

**Solution**:

. Verify you are logged in: `oc whoami`
. Check your current project: `oc project`
. Confirm you are using the correct namespace: `oc project {namespace}`
. If you still see Forbidden errors, your user may not have been provisioned correctly. Check that your username matches the one shown in the lab credentials panel.

---

**Issue**: The web terminal does not appear when clicking the `>_` icon

**Solution**:

* Wait up to 60 seconds for the terminal pod to start on first use
* Refresh the browser page and try again
* Clear browser cache and cookies, then log in again
* Try a different supported browser (Chrome or Firefox recommended)

---

**Issue**: The `{namespace}` project is not visible

**Solution**:

. Run `oc get projects` to list all projects you have access to
. If `{namespace}` is not listed, your environment may still be provisioning. Wait 2-3 minutes and try again.
. If the project is still missing after 5 minutes, contact your lab instructor.
