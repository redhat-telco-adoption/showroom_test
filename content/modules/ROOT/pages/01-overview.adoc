== Your role and mission

It is Monday morning at ACME Corp. You receive an urgent message from your manager, Jordan:

____
"Hey, Alex went on vacation last Friday and left several deployments in a broken state. I'm starting to get calls from the business about services being down. Can you log into the cluster and start investigating? Standup is at 9 AM."
____

You are a senior SRE at ACME Corp. ACME recently completed a migration of their e-commerce platform to Red Hat OpenShift {ocp_version}, and the team is still building confidence with the new platform. Today, your job is to work through a series of broken deployments, diagnose what went wrong, and restore each application to a running state.

Each module in this workshop presents a realistic failure scenario based on common OpenShift problems that developers and SREs encounter in production.

== Your mission

By completing this workshop, you will have proven that you can:

* Access an OpenShift cluster using both the `oc` CLI and the web console
* Diagnose failing pods using events, logs, and describe output
* Fix configuration problems involving ConfigMaps, Secrets, and environment variables
* Resolve storage-related failures involving PersistentVolumeClaims
* Troubleshoot RBAC, SecurityContextConstraints, and image pull authentication issues
* Debug networking problems including misconfigured services, routes, and network policies
* Address resource constraint failures including OOM kills and quota exceeded errors
* Resolve application-level failures including CrashLoopBackOff and liveness probe failures

== Target audience

This workshop is designed for:

* *Developers* who deploy applications on OpenShift and want to debug their own workloads independently
* *Site reliability engineers* who need to triage and resolve production incidents on OpenShift clusters
* *Platform engineers* who support development teams and respond to cluster support requests

All experience levels are welcome. Module 1 provides foundational skills, and experienced practitioners can start at any module.

== ACME Corp's situation

**The platform**: ACME Corp runs a microservices-based e-commerce platform on a Red Hat OpenShift {ocp_version} multinode cluster.

**The team**: A small platform engineering team supports 15 developers across 3 squads. Developers deploy their own applications but sometimes run into issues they cannot resolve on their own.

**The challenge**: Alex, a junior developer, deployed updates to multiple services before going on vacation. Several configurations were incomplete or incorrect, leaving applications in failed states. The platform team needs to investigate each failure, identify the root cause, and apply a fix.

**Your goal**: Work through each module, fix the broken deployment, and understand why it failed so you can prevent similar issues in the future.

== What success looks like

At the end of each module, you will:

. Identify the root cause of the failure from logs and events
. Apply the correct fix using `oc` CLI commands
. Verify that the application returns to a `Running` state
. Understand the concept behind the fix so you recognize the pattern next time

== Common questions

**"Do I need to complete every module?"**

No. Each module is self-contained. You can start at any module and skip areas you are already comfortable with.

**"Will I break anything by experimenting?"**

No. Each module has its own isolated namespace. You can run any `oc` commands you like without affecting other learners or the broader cluster.

**"What if I get stuck?"**

Each module includes a troubleshooting section with solutions to common problems. You can also refer to the xref:99-cheatsheet.adoc[Troubleshooting Cheatsheet] at any time.
