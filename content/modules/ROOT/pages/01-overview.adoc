Welcome to the *OpenShift Troubleshooting Workshop*. In this workshop you take on the role of a platform engineer or developer responding to a series of production incidents on an OpenShift cluster. 19 applications are broken -- each one with a different root cause. Your job is to diagnose and fix them all.

== Your challenge

A team of developers has deployed 19 applications across 6 namespaces. Every application has a deliberate defect representing a failure mode commonly encountered in production environments. Some pods refuse to start, some crash immediately, some run but cannot serve traffic, and some are stuck waiting for resources that do not exist.

Each lab presents one broken application. You will use the `oc` CLI and the OpenShift Web Console to investigate the symptoms, identify the root cause, and apply a fix. The application must return to `Running` state and serve traffic through its Route.

== What you will learn

By completing this workshop you will be able to:

* Diagnose configuration failures caused by missing ConfigMaps and Secrets
* Resolve storage issues including missing PVCs, wrong StorageClasses, and volume permission errors
* Fix security violations involving RBAC, SCCs, and image pull authentication
* Troubleshoot networking failures including selector mismatches, route misconfiguration, DNS blocking, and NetworkPolicy issues
* Identify and correct resource constraint problems including scheduling failures, OOMKills, and quota exhaustion
* Resolve application lifecycle problems including bad command overrides, misconfigured probes, and wrong image tags

== Target audience

This workshop is designed for:

* *Platform engineers* managing OpenShift clusters and supporting developer teams
* *Developers* deploying and operating applications on OpenShift
* *Site reliability engineers (SREs)* responding to application incidents
* Anyone preparing for OpenShift administrator or developer certifications

A basic familiarity with Kubernetes concepts (Pods, Deployments, Services) is helpful but not required.

== How the labs are structured

Each lab follows the same structure:

. *Overview* -- describes the broken application and your mission
. *Learning objectives* -- the specific skills you will practice
. *Exercise 1: Investigate the problem* -- guided steps to explore symptoms and diagnose the root cause using both the Web Console and the `oc` CLI
. *Exercise 2: Resolve the issue* -- steps to apply a fix and verify that the application is healthy

Progressive hints are available in each lab. Try to work through the problem before opening them. A full solution is available for instructors.
